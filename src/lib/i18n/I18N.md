# Internationalization (EN/FR)

This site uses `react-i18next` with bundled JSON under `src/locales/{en,fr}/`.

## Routing (SEO)

- **English** stays at unprefixed URLs (`/about-us`, `/services`, вЂ¦).
- **French** is path-based under `/fr/...` (`/fr/about-us`, `/fr/services`, вЂ¦).
- Each locale tree has its own root layout (`app/(en)/`, `app/(fr)/`) so static HTML gets the correct `html[lang]`, localized `<title>`/`description`, and `hreflang` alternates.
- The EN/FR switcher navigates between the two path trees (and still mirrors the choice into `localStorage` for convenience).

UI body copy still hydrates through `react-i18next` from the active locale path. Prefer path locale over any stored preference when both disagree.

## Rules for new copy
1. Do **not** hardcode user-facing UI strings in components.
2. Add the English key to `src/locales/en/<namespace>.json`.
3. Add the same key to `src/locales/fr/<namespace>.json` (draft FR is fine pending review).
4. Read it with `useTranslation("<namespace>")` and `t("key.path")`.
5. Run `npm run i18n:check` (also runs in `npm run build`) вЂ” missing FR keys fail the build.
6. **Do not translate** proper nouns: project names, brand/partner names, people names, or place names (e.g. вЂњValley Park LibraryвЂќ, вЂњToronto, ONвЂќ). Keep those in the source data as English.
7. Page SEO `meta.title` / `meta.description` live in the page namespace JSON (and home `meta` in `home.json`). Wire them through `buildPageMetadata` / `namespacePageMetadata` in `src/lib/i18n/seo.ts`, and add a matching `/fr/...` page when you add a new public route.

## Namespaces
- `common` вЂ” nav, footer, cookies, intro splash, sector nav labels
- `home` вЂ” homepage sections
- `about` вЂ” About Us page
- `contact` вЂ” Contact page + inquiry/quote form
- `services` вЂ” Services index chrome, category/card labels, detail chrome, default process
- `serviceDetails` вЂ” per-slug overview / bestFor / deliverables / FAQs
- `sectors` вЂ” Sectors index, market hubs (`/markets/[slug]`), market/sub-sector card copy
- `sectorDetails` вЂ” deep `/sectors/[slug]` page bodies
- `portfolio` вЂ” Portfolio index + project detail chrome/scopes/specs (names & locations stay EN)
- `careers` вЂ” Careers page + application form
- `insight` вЂ” Insight / LinkedIn newsroom page (`/linkedin`)
- `legal` вЂ” Legal Centre chrome + per-doc titles (full policy bodies still placeholders)
- `store` вЂ” Store UI chrome (nav, filters, cart/checkout, category labels, PDP chrome, empty/status)
- `storeCatalog` вЂ” Store product body fields (description, material/finish, delivery, specs, sample reviews). Product **names** stay EN in `products.ts`
- `capabilityPdf` вЂ” Capability PDF viewer chrome + shared company blocks / section labels
- `capabilityPdfDetails` вЂ” Per-sub-sector capability statement body (keyed by slug)

Capability PDF body FR drafts may be refined later; chrome/shared are fully bilingual. Full legal document bodies should be counsel-owned when they replace placeholders.
